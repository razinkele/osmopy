# UI Improvements Design Spec

**Date:** 2026-03-12
**Scope:** 7 usability improvements to the OSMOSE Shiny UI

---

## 1. Example Loading — Persistent Header Bar

**Problem:** The example dropdown resets after loading, losing context about which config is active.

**Design:**
- Keep the dropdown selector in Setup page, add an explicit **"Load"** button next to it
- After loading, show a **persistent header bar** at the top of the content area (visible across all tabs):
  - Left: config name in amber (e.g., "Eec Full"), species count, computed parameter count via `len(state.config.get())`
  - Right: "modified" indicator when config has been changed since load
- The header bar **replaces** the existing `dirty_banner` — both show modification state, so the header bar subsumes it
- The header bar reads from `state.config_name` (new reactive value) and `state.dirty`
- Dropdown retains the selected value after load (no reset to empty)

**State changes:**
- Add `state.config_name: reactive.Value[str]` — set on example load or config import
- Remove `state.config_param_count` — compute inline as `len(state.config.get())` in the header render
- Header bar rendered in `app.py` as `output_ui("config_header")`, placed above the `navset_pill_list`, replacing `dirty_banner`

**Files:** `ui/state.py`, `app.py`, `ui/pages/setup.py`

## 2. Species Configuration — Spreadsheet Table

**Problem:** The accordion-per-species layout is huge and unmanageable with 8+ species.

**Design:**
- Replace the per-species accordion with a **spreadsheet-style table**:
  - Rows = parameters (e.g., "L-infinity", "Growth coeff K")
  - Columns = species (e.g., "Anchovy", "Sardine", "Hake")
  - Cells = inline editable `input_numeric` / `input_text` / `input_select` depending on field type
- Rows grouped by category with **collapsible group headers** (Growth, Reproduction, Predation, Mortality, Fishing, Initialization)
  - Group header row spans full width, shows category name + param count
  - Click to expand/collapse (default: Growth expanded, others collapsed)
- Parameter name column is sticky (stays visible on horizontal scroll)
- Units shown in muted text next to parameter name
- Validation errors shown as red cell borders with tooltip on hover
- "Show Advanced" toggle hides/shows advanced parameters (e.g., growth class name, threshold age)
- **When `nspecies == 0`**, show a placeholder message: "Load a configuration to view species parameters"

**Input ID namespacing:**
- Table cell IDs use prefix `spt_` to avoid collision with IDs generated by `render_field()` elsewhere
- Pattern: `spt_{param_key_without_idx}_{species_idx}` (e.g., `spt_species_linf_0`, `spt_species_k_1`)
- This is distinct from the existing `render_field()` pattern (`species_linf_sp0`)

**Sync strategy:**
- Each cell change triggers a single `state.update_config()` call for that cell's key only
- Uses a per-cell `@reactive.effect` watching the cell's input value (same pattern as existing `sync_inputs()` but scoped to one key)
- `state.loading` guard prevents sync during bulk config load

**Implementation approach:**
- New function `render_species_table()` in `ui/components/param_form.py`
- Generates an HTML table with Shiny `input_numeric` / `input_text` widgets embedded in cells
- Category collapse state managed client-side with JS (no server round-trips)

**Files:** `ui/components/param_form.py`, `ui/pages/setup.py`

## 3. LTL/Plankton Configuration — Same Spreadsheet

**Problem:** LTL resource parameters use the same verbose per-species layout.

**Design:**
- Apply the identical spreadsheet table pattern from §2 to LTL/resource species
- Resource species are indexed from `nspecies` to `nspecies + nresource - 1`
- `nresource` comes from config key `simulation.nresource` (same convention as `simulation.nspecies`)
- Reuse `render_species_table()` with a `fields` parameter to pass LTL-specific fields and `start_idx=nspecies`
- Placed in the Forcing page's "Lower Trophic Level" section

**Files:** `ui/pages/forcing.py`, `ui/components/param_form.py`

## 4. Grid Preview — Spatial File Overlay Selector

**Problem:** Grid preview only shows the grid extent, not other spatial data referenced in config.

**Design:**
- Add a **dropdown above the map** listing all spatial files referenced in the config
- Build choices by scanning the schema registry for all fields with `param_type == ParamType.FILE_PATH`, then checking which resolved keys exist in the current config and reference `.nc` or `.csv` files
- Choices labeled by field description + species name where applicable:
  - "Grid extent" (default, always present)
  - "Reproduction season: Anchovy" (from `reproduction.season.file.sp0`)
  - "Movement map: Sardine" (from movement-related FILE_PATH fields)
  - etc.
- Selecting a file loads and overlays it on the map:
  - **NetCDF files:** Read via xarray, auto-detect the first 2D numeric variable, render as colored polygon grid cells using the file's lat/lon coordinates
  - **CSV files:** Expect columns with spatial index or lat/lon + value; render as colored grid cells mapped to the base grid
  - **Missing files:** Show notification "File not found: {path}" and skip overlay
- Grid extent layer always visible underneath overlays

**Files:** `ui/pages/grid.py`

## 5. Results Directory — Smart Defaults + Auto-load

**Problem:** Results page defaults to non-existent `output/` directory.

**Design:**
- Default output path: `state.output_dir.get()` if set (from last run), otherwise empty string
- Replace the "Browse" modal concept with a simpler approach: **text input + validation message**
  - If the path doesn't exist, show red text: "Directory not found"
  - If it exists but contains no OSMOSE output files, show: "No results found in this directory"
  - If valid, show green text: "Found N output files"
- **Auto-load:** Watch `input.main_nav()` in a `@reactive.effect`; when value is `"results"` and `state.output_dir` is set and results are not already loaded, trigger loading automatically
  - Add `state.results_loaded: reactive.Value[bool]` (default `False`) — set to `True` after loading, reset to `False` when `state.output_dir` changes
  - Prevents redundant re-loads on every tab switch
- "Load Results" button remains for manual reload or loading a different directory
- When loading from a completed run, `state.output_dir` is already set by the run handler — auto-load picks it up

**Files:** `ui/pages/results.py`, `ui/state.py`

## 6. Species Filter — Populate from Config

**Problem:** Species filter dropdown only shows "All Species" regardless of loaded config.

**Design:**
- Add `state.species_names: reactive.Value[list[str]]` to `AppState`
- Populate on every config load/change by extracting all resolved keys matching the `species.name.sp{idx}` field pattern from the schema registry for indices `0..nspecies-1`
- Any page with a species filter (Results, Grid overlays) reads from `state.species_names`
- Filter always includes "All Species" as first option, followed by individual species names
- On Results page, species filter also cross-references with available data columns (some species may have no output)

**Files:** `ui/state.py`, `ui/pages/results.py`, `ui/pages/setup.py`

## 7. Tooltips — Layered Help System

**Problem:** Many parameters and options lack explanatory text. Fields like "Path to restart file" are unclear without context.

**Design:**
- **Hover tooltips:** Add a `(?)` icon next to field labels. Hover shows Bootstrap popover with:
  - Field description (from `OsmoseField.description`)
  - Valid range and unit (from `min_val`, `max_val`, `unit`)
  - Default value
  - OSMOSE config key name (for advanced users)
- **Global "Show Help" toggle:** Button in the global `osmose-header` div in `app.py`, next to the existing About/Help buttons. When active:
  - All field descriptions appear as muted text below each input (inline)
  - Toggle state persisted in `localStorage` via JS
  - Adds CSS class `show-all-help` on `<body>`, with CSS rules to show/hide `.field-help-text` elements
- **Non-schema fields** (Java options, timeout, output dir, etc.): add manual tooltip text as a dict in `ui/tooltips.py`

**Bootstrap popover initialization:**
- Add JS snippet in `app.py` head scripts to initialize all Bootstrap popovers:
  ```js
  document.addEventListener('shiny:value', function() {
      document.querySelectorAll('[data-bs-toggle="popover"]').forEach(function(el) {
          if (!el._bsPopover) new bootstrap.Popover(el);
      });
  });
  ```
- Re-initializes after every Shiny output update to catch dynamically rendered popovers
- For the spreadsheet table, tooltips go on the parameter name cell (first column)

**Files:** `ui/components/param_form.py`, `ui/tooltips.py` (new), `www/osmose.css`, `app.py`

## Cross-cutting Concerns

**Reactive patterns:**
- All new reactive values use `reactive.isolate()` when reading config inside effects (per existing pattern)
- Spreadsheet table cell changes trigger per-cell `state.update_config()` (not bulk sync)
- `state.loading` guard prevents sync effects during bulk config load

**Performance:**
- Spreadsheet table with 30 params x 8 species = 240 cells — well within Shiny's capacity
- Category collapse is client-side JS only (no server round-trip)
- Grid overlay loading is on-demand (only when selected from dropdown)

**Theme compatibility:**
- All new components respect dark/light theme via existing CSS variables
- Spreadsheet table uses same color palette as existing UI (`#0f1923`, `#162232`, `#2d3d50`, `#d4a017`)

**Testing:**
- Unit tests for `render_species_table()` with 0, 1, and 8 species
- Unit test for config header bar rendering (with and without config loaded)
- Unit test for species name extraction from config
- Test that tooltip markup is generated correctly for schema fields
- Test auto-load logic: results loaded flag, tab switch trigger
