Walk through adding a new OSMOSE parameter to the schema-driven architecture.

## Arguments

- `name` (required): the parameter name (e.g., "species.egg.size")
- `type` (optional): field type — "float", "int", "bool", "str", "file" (default: "float")

## Steps

1. **Identify the schema module**: Find which file in `osmose/schema/` the parameter belongs to. Group by process:
   - `species.py` — species-level biological parameters
   - `predation.py` — predation-related parameters
   - `mortality.py` — natural/fishing mortality
   - `reproduction.py` — reproduction/recruitment
   - `movement.py` — spatial movement maps
   - `grid.py` — spatial grid configuration
   - `simulation.py` — simulation control parameters
   - `output.py` — output configuration

2. **Check for existing similar fields**: Search for related parameters:
   ```
   grep -r "{param_prefix}" osmose/schema/
   ```

3. **Add the OsmoseField**: Add the field definition to the appropriate schema class. Follow existing patterns in the file:
   - `key_pattern`: lowercase dot-separated OSMOSE key (e.g., `species.egg.size.sp{idx}`)
   - `label`: human-readable label for UI
   - `description`: tooltip text explaining the parameter
   - `default`: default value (match Java defaults)
   - `min_val`/`max_val`: validation bounds if applicable
   - For species-indexed params, include `sp{idx}` placeholder in `key_pattern`

4. **Verify UI rendering**: The UI auto-generates forms from schema metadata. Confirm:
   ```
   .venv/bin/python -c "from osmose.schema import registry; print([f for f in registry.all_fields() if '{name}' in f.key_pattern])"
   ```

5. **Verify config round-trip**: Test that the parameter reads from and writes to OSMOSE config format:
   ```
   .venv/bin/python -m pytest tests/test_schema.py -v -k "round_trip or registry"
   ```

6. **Run full test suite** to check for regressions:
   ```
   .venv/bin/python -m pytest tests/ -x -q
   ```

## Rules

- Always use `.venv/bin/python`, never system python
- Run from `/home/razinka/osmose/osmose-python/`
- Match Java OSMOSE 4.3.3 key naming conventions exactly
- Config keys are case-sensitive — check Java source if unsure
- Species-indexed params use `sp{idx}` placeholder, resolved via `field.resolve_key(idx)`
- Input IDs in Shiny are generated by `key_pattern.replace(".", "_")` — check for collisions with existing fields
